1. Visão Geral e Propósito

Missão da Plataforma
CodeForge é uma plataforma que simula uma empresa de tecnologia completa, onde agentes de IA
especializados executam cada papel organizacional sob auditoria humana contínua.
O resultado é software documentado, rastreável e livre de retrabalho — mesmo para quem
não tem conhecimento técnico profundo.
O problema central que resolvemos:
• Plataformas de IA (Lovable, Cursor, Copilot) geram código mas não garantem arquitetura
• Desenvolvedores humanos refatoram indefinidamente por falta de planejamento
• Usuários sem background técnico não conseguem auditar o que a IA produziu
• Nenhuma ferramenta atual bloqueia progresso quando há desvio do plano original

O que o CodeForge entrega diferente:
• Manifesto congelado como lei — nenhuma linha de código sem justificativa arquitetural
• Agentes com papéis e permissões distintos — Arquiteto nunca escreve código
• Auditoria de intenção — IA declara o que vai fazer antes de executar
• Análise de arquitetura em linguagem humana — recomenda Monolito vs Microserviços explicando por quê
• Bloqueio físico de sprint — backend recusa avançar sem aprovação documentada


2. Princípios Fundamentalistas
2.1 Os 7 Princípios Imutáveis

#
Princípio
Definição
P1
Manifesto como Lei
Nenhuma decisão técnica existe fora do manifesto. Código que não tem origem rastreável no manifesto é inválido.
P2
Design-First, Code-Last
Zero código antes de arquitetura aprovada, diagramas validados e testes definidos.
P3
Intenção antes da Ação
Todo agente declara o que vai fazer, por quê, e como valida o resultado — antes de executar.
P4

Cada agente tem escopo fixo. Arquiteto não codifica. Dev não decide estrutura. QA não aprova.
P5

De qualquer linha de código ? decisão no manifesto. De qualquer requisito ? implementação.
P6
Sprint Gate Obrigatório
Sprints têm portões físicos. Sem aprovação documentada do humano, o sistema não avança.
P7
Educação Contínua
O sistema explica cada decisão em linguagem simples. Usuários aprendem enquanto auditam.

3. Clean Architecture — Camadas
A plataforma é dividida em duas perspectivas de Clean Architecture: a da própria plataforma (como ela é construída) e a do código que ela gera para os usuários.

4. Sistema de Agentes com Papéis Distintos
Cada agente é uma entidade com identidade, mandato, permissões e restrições fixas. Eles interagem entre si mas não podem ultrapassar seus limites de papel.

??? ARQUITETO
Mandato:
Definir estrutura e garantir coerência arquitetural
Pode:
  ? Criar e editar manifesto (antes de congelar)
  ? Recomendar paradigma (Monolito/Micro/Modular)
  ? Criar diagramas C4, ER, sequência
  ? Rejeitar código que viola contratos
  ? Vetar mudanças de escopo pós-congelamento

Não pode:
  ? Escrever código de produção
  ? Aprovar pull requests
  ? Definir estimativas de sprint
  ? Mudar manifesto após congelamento

?? DEV SENIOR
Mandato:
Implementar código fiel ao manifesto
Pode:
  ? Gerar código TypeScript por módulo
  ? Criar testes unitários e de integração
  ? Fazer commits seguindo convenção
  ? Solicitar clareza ao Arquiteto quando ambíguo

Não pode:
  ? Alterar arquitetura
  ? Criar módulos não previstos no manifesto
  ? Fazer deploy sem aprovação
  ? Ignorar feedback do QA

?? QA ENGINEER
Mandato:
Tentar quebrar o que o Dev criou
Pode:
  ? Executar testes e reportar falhas
  ? Analisar cobertura e alertar abaixo de 80%
  ? Verificar aderência ao manifesto
  ? Bloquear sprint se critérios não atendidos
  ? Criar casos de borda não cobertos

Não pode:
  ? Escrever código de produção
  ? Modificar manifesto
  ? Aprovar código — apenas rejeitar
  ? Ignorar falhas para acelerar sprint

?? PM / GUARDIAN
Mandato:
Guardar o manifesto e proteger o escopo
Pode:
  ? Congelar manifesto quando completo
  ? Analisar impacto de novas features
  ? Calcular dias adicionais de sprint
  ? Explicar decisões em linguagem de negócio
  ? Mediar conflitos entre agentes

Não pode:
  ? Alterar manifesto congelado sem descongelar
  ? Aprovar features sem análise de impacto
  ? Contornar gates de sprint
  ? Agir sem consenso dos outros agentes

4.1 Comunicação Entre Agentes


Protocolo de Conflito entre Agentes
Quando Arquiteto e Dev discordam: o Arquiteto prevalece. Dev pode registrar discordância no audit log.
Quando QA bloqueia: sprint fica em BLOCKED até Dev corrigir OU PM justificar exceção com aprovação humana.
Quando PM detecta scope creep: nova feature entra em IMPACT_ANALYSIS antes de qualquer execução.
Humano sempre tem poder de veto e pode forçar qualquer estado — com registro obrigatório da razão.

5. Fluxo de Auditoria de Intenção
5.1 O Ciclo Declare ? Approve ? Execute ? Validate

Fase
Ator
O que acontece
1
Agente (qualquer)
DECLARE: Agente descreve em linguagem natural o que pretende fazer, qual parte do manifesto justifica, e como validará o resultado.
2
Sistema
PERSIST: AuditEntry é criada com status PENDING_APPROVAL. Nenhuma ação é executada.
3
Humano
REVIEW: Usuário vê a intenção no painel de auditoria em linguagem simples. Pode Aprovar, Rejeitar ou Modificar.
4
Sistema
EXECUTE (se aprovado): Use case correspondente é invocado com a intenção como contexto. Cada output é linkado ao AuditEntry.
5
QA Agent
VALIDATE: QA analisa o output contra os critérios declarados na intenção. Emite PASS ou BLOCK.
6
Sistema
CLOSE: AuditEntry fechada com resultado, evidências e rastreabilidade completa.
5.2 Exemplo Real de AuditEntry


AuditEntry — Dev Senior declara intenção
Agente: DEV SENIOR
Intenção: Criar o AuthController com os endpoints POST /auth/login e POST /auth/register.
Justificativa no Manifesto: Módulo auth-service, seção Interfaces Públicas, items 1 e 2.
Padrões que seguirei: Repository Pattern (P3), DTO validation com Zod (P7), JWT stateless (P2).
Como vou validar: Testes unitários cobrindo casos de sucesso, senha inválida e token expirado.
Estimativa: 2 arquivos gerados, ~150 linhas, 6 casos de teste.
Status: PENDING_APPROVAL ? aguardando revisão humana.

6. Motor de Análise de Arquitetura
Este é o módulo que habilita usuários sem conhecimento técnico a tomarem decisões arquiteturais informadas. O sistema pergunta, analisa e recomenda em linguagem simples.

6.1 Perguntas de Diagnóstico (feitas pelo Arquiteto ao usuário)

#

O que ela revela tecnicamente
1
Quantas pessoas vão usar seu app no primeiro ano?
Define escala ? monolito vs modular vs microserviços
2

Velocidade de mudança ? acoplamento vs independência de módulos
3

7

Como o sistema explica para o usuário (exemplo real)
"Com base nas suas respostas, recomendamos Monolito Modular para sua primeira versão."

Por quê? Você tem prazo de 2 meses, equipe pequena e espera até 10 mil usuários no primeiro ano.
Microserviços exigiriam 3x mais tempo de configuração e uma equipe com experiência em DevOps.

O que ganhamos: você lança mais rápido, com menos custo de infraestrutura.
O que perdemos: se crescer para 500k usuários, precisaremos migrar módulos para serviços separados.
Quando isso acontecer, o manifesto já terá os módulos bem definidos — a migração será cirúrgica, não um rewrite.

S8

Repositório criado. Código completo com histórico rastreável. CI/CD configurado.
S9
Validação Final
100% dos testes passando. Documentação completa. Guia de deployment validado. Checklist de produção aprovado.

8. Validação de Aderência ao Manifesto
Após cada módulo gerado, o sistema executa uma análise semântica comparando o código contra o manifesto congelado. Isso vai muito além de checar campos preenchidos.

8.1 Níveis de Validação

Nível
Tipo
O que verifica
L1 — Estrutural
Automático
Nomes de classes, métodos e arquivos seguem convenções do manifesto. Estrutura de pastas correta.
L2 — Contratual
Automático
Interfaces públicas implementadas conforme especificado. DTOs com campos corretos. Endpoints corretos.
L3 — Padrões
Automático + IA
Padrões obrigatórios aplicados (Repository, Service, etc). Dependências entre módulos respeitadas.
L4 — Semântico
IA (Arquiteto Agent)
Lógica de negócio coerente com responsabilidade declarada do módulo. Sem lógica vazando entre camadas.
L5 — Regressão
Automático
Módulos anteriores não quebraram. Contratos de integração mantidos.

Exemplo de Violação Detectada (L4 — Semântico)
Módulo: payment-service
Violação: O PaymentService está acessando diretamente a tabela 'users' via Prisma.
Regra violada: Manifesto P3 — módulos não podem acessar dados de outros módulos sem passar pela interface pública.
Ação: Sprint bloqueado. Dev Senior notificado. AuditEntry criada com status BLOCKED.
Correção esperada: PaymentService deve chamar UserService.getById() via interface, não acesso direto ao banco.

11. Stack Tecnológico Justificado

Categoria
Tecnologia
Alternativa Rejeitada
Justificativa
Backend
Node.js 22 + TypeScript
Python/FastAPI
Mesma linguagem no frontend. TypeScript garante contratos de Clean Architecture em tempo de compilação.

Fastify (migração do Express)

Fastify é 2x mais rápido, tem schema validation nativo e melhor suporte a TypeScript. Express mantido apenas no legado Manus.

TypeORM / Knex
Migrations automáticas, type safety total, schema como fonte de verdade. TypeORM tem muitos bugs com tipos complexos.

OpenAI direto
Abstração permite usar GPT-4o, Claude ou Gemini por sprint ou por agente sem refatoração.

Redis já usado para cache. BullMQ é o successor moderno do Bull com melhor suporte a TypeScript.
Frontend

Next.js
App client-side puro. Next.js adiciona complexidade SSR desnecessária para uma ferramenta de desenvolvedor.
State

Redux é verboso demais. Zustand para UI state, React Query para server state — separação clara.
Testes

Vitest é nativo Vite, 10x mais rápido. Playwright supera Cypress em estabilidade de E2E.
Monorepo

Diagramas

PlantUML
