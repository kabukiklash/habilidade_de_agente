# AI-Assisted Platform Builder

This skill guides the development of complex software platforms using a structured 9-sprint workflow with AI (GPT-4o) assistance. It ensures consistency, rastreability, and quality through manifesto-driven development.

## When to Use This Skill

Use this skill when building complex platforms (runtimes, kernels, compilers, frameworks), requiring AI assistance for architecture and code generation, needing complete rastreability of decisions and implementations, wanting to minimize rework through upfront planning, or developing applications where consistency is critical.

## Core Principles

**Manifesto as Single Source of Truth**: All decisions are documented in a frozen manifesto that guides implementation. Before each action, consult the manifesto to ensure alignment.

**Design-First, Code-Last**: Complete architecture, design, and planning before code generation. This prevents rework and ensures consistency.

**Rastreability**: Every decision includes alternatives considered and justification. This enables quick resolution of issues and maintains project coherence.

**Autonomous AI**: The AI (GPT-4o) makes architectural decisions (modular vs monolithic, REST vs gRPC, etc.) based on analysis, not just following instructions.

**Continuous Validation**: At each sprint boundary, validate against the manifesto. Identify and correct deviations immediately.

## The 9-Sprint Workflow

The workflow consists of 9 sequential sprints, each 1-3 days. Sprints 1-2 focus on research and architecture to understand the domain and define high-level structure. Sprints 3-4 involve AI-driven planning with chat refinement, manifesto generation, and completeness validation. Sprints 5-7 handle implementation including UML diagrams, TypeScript code, Docker environment, and GitHub integration. Sprints 8-9 focus on validation and deployment with tests, documentation, and production preparation.

See `references/9-sprint-workflow.md` for detailed sprint descriptions.

## Getting Started

### Step 1: Initialize the Monorepo

Use the provided script to create the project structure:

```bash
python /home/ubuntu/skills/ai-assisted-platform-builder/scripts/init_monorepo.py <project-name>
```

This creates backend package (Node.js + Express + TypeScript), frontend package (React + Vite + TypeScript), documentation directory, and GitHub Actions workflow directory.

### Step 2: Create the Manifesto

Use the manifesto template to document your platform's vision:

```bash
cp /home/ubuntu/skills/ai-assisted-platform-builder/templates/MANIFESTO_TEMPLATE.md <project>/MANIFESTO.md
```

Fill in all sections: identity, objectives, modules, architecture, decisions, implementation rules, observability, security, performance, deployment, and roadmap.

### Step 3: Follow the 9-Sprint Workflow

Execute each sprint in order. Sprint 1 researches best practices and patterns in your domain. Sprint 2 creates architecture diagrams (C4, ER, sequence). Sprint 3 chats with GPT-4o to extract and validate concepts. Sprint 4 generates and freezes the manifesto. Sprint 5 generates UML diagrams (4+ types). Sprint 6 generates TypeScript code with tests. Sprint 7 creates Docker environment and GitHub integration. Sprint 8 executes tests and creates documentation. Sprint 9 prepares for production deployment.

### Step 4: Consult the Manifesto Continuously

Before each major decision or code generation, read the manifesto, validate that the proposed action aligns with documented decisions, document any new decisions with justification, and update the manifesto if needed (before freezing).

## Key Artifacts

**Manifesto**: Frozen document containing all decisions, architecture, rules, and rationale. Never deviate without documenting why.

**Architecture Diagrams**: C4 (context, container, component), ER, sequence, and dependency diagrams generated from manifesto.

**Generated Code**: TypeScript controllers, services, repositories, types, and tests generated by AI based on manifesto.

**Docker Compose**: Complete environment with backend, frontend, PostgreSQL, Redis, and other services.

**GitHub Repository**: Automated creation and push of code with CI/CD workflows.

**Documentation**: Technical docs, deployment guide, user guide, and API reference.

## Best Practices

Freeze the manifesto early once you have sufficient information. This prevents scope creep and keeps the AI focused. Validate at checkpoints before each sprint to ensure implementation aligns with the manifesto. Use AI for decisions by letting GPT-4o analyze trade-offs and recommend architectures, documenting its reasoning. Document everything including every decision, alternative, and justification in the manifesto or decision history. Generate tests alongside code (TDD) aiming for 80%+ coverage. Iterate quickly with each sprint being 1-3 days, adjusting if something isn't working.

## Common Pitfalls to Avoid

**Incomplete Manifesto**: Rushing through manifesto creation leads to ambiguities and rework. Invest time upfront.

**Scope Creep**: Freeze the manifesto before implementation. New features require a new sprint cycle.

**Ignoring Manifesto**: Consulting the manifesto at every step prevents drift and maintains consistency.

**Skipping Validation**: Validating at sprint boundaries catches issues early when they're cheap to fix.

**Over-Engineering**: The manifesto should guide simplicity. Avoid adding unnecessary complexity.

## Troubleshooting

If code doesn't match the architecture, consult the manifesto as the AI may have deviated. Correct and regenerate. If you forgot why a decision was made, check the decision history in the manifesto where all decisions should have justification. If the project is taking too long, the likely cause is an incomplete manifesto or scope creep. Freeze the manifesto and stick to it. If tests are failing, regenerate tests based on the latest code and ensure the manifesto's validation rules are being applied.
